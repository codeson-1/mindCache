# 面试追问：搜索召回「宁缺毋滥」方案

## 场景

面试官看到你的项目用了三路融合检索（向量 + BM25 + 时间衰减），追问：

> "你搜 spring，topK 设置 20，但库里只有 4 条笔记。其中一条内容是 '12'、跟 spring 完全无关，为什么会被搜出来？你怎么解决？"

---

## 问题分析

### 根因

当前检索流程中，三个环节导致了无关结果的漏网：

```
search("spring", topK=20)
 ├─ pgvector 语义召回：similarityThresholdAll() — 不做任何过滤，全部 4 条都进入候选
 ├─ Lucene BM25 召回：Candidate 数 ≤ 4，min-max 归一化后最小分也是 0.0
 └─ 融合评分：fusedScore = 0.5×0.0 + 0.3×0.0 + 0.2×1.0 = 0.20（纯靠时间衰减）
    → limit(20) 直接返回，不关心分数有多低
```

简单说：**topK 大于总笔记数时，所有候选都被返回，不管多不相关**。"12" 的 fusedScore 只有 0.20（全部来自时间衰减），仍然出现在结果里。

### 融合分数拆解（搜 "spring" 时各笔记得分）

| 笔记 | 向量归一化 | BM25归一化 | 时间衰减 | fusedScore | 判断 |
|------|-----------|-----------|---------|-----------|------|
| Spring AI 相关知识 | 1.0 | 0.10 | 1.0 | **0.73** | ✅ 相关 |
| Spring Boot 3.5 笔记 | 0.77 | 1.0 | 0.97 | **0.88** | ✅ 相关 |
| Spring Boot 虚拟线程 | 0.36 | 0.0 | 0.97 | **0.38** | ⚠️ 弱相关 |
| 12 | 0.0 | 0.0 | 1.0 | **0.20** | ❌ 无关 |

"12" 的 0.20 分全部来自 `gamma × timeDecay = 0.2 × 1.0`，没有任何语义或关键词贡献。

---

## 方案对比

### ❌ 方案一：固定阈值过滤（单层方案）

在融合后加一个绝对值阈值，如 `fusedScore < 0.25` 就过滤。

```
优点：实现最简单，1 行代码
缺点：
  • 绝对值阈值对不同 query 不稳定——有的场景所有结果都在 0.3~0.4，全被误杀
  • 对 BM25 漏网无能为力（阈值设在融合层，BM25 命中但语义无关的仍有较高分）
  • 需要针对实际数据反复调参
```

### ✅ 方案二：双重过滤（采用方案）

两层互补：

```
第一道 — pgvector 索引层（数据库侧）
  similarityThreshold(0.65)：在 HNSW 索引扫描时即过滤，不相关向量不进入候选集
  → 拦截「语义无关」的（如 "12" 搜 "spring"）

第二道 — 融合层相对分差截断（应用层）
  fusedScore < top1 × 0.75 → 过滤
  → 拦截「BM25 碰巧命中但语义无关」的漏网之鱼
```

#### 为什么选这个

1. **改动小**：SearchConfig +2 配置项，SearchService 约 15 行，不臃肿
2. **自适应**：相对分差用 top1 作为基准，不写死绝对值——最佳 0.9 时截断线 0.675，最佳 0.5 时截断线 0.375，小库场景不轻易空结果
3. **分层解耦**：向量过滤下推到数据库索引层（不是你写的 Java 循环），BM25 漏网靠融合层兜底
4. **可调参**：两个参数都在 `application.yml` 里，评测时不用改代码

## 实现代码

### SearchConfig.java（新增配置项）

```java
@ConfigurationProperties(prefix = "mindcache.search")
public class SearchConfig {

    // ... 原有 alpha/beta/gamma ...

    /**
     * pgvector 向量检索的最低余弦相似度阈值（0~1）。
     * 低于此值的向量不会进入候选集，在数据库/HNSW 层即可拦截语义无关的结果。
     * 默认 0.65。
     */
    private double similarityThreshold = 0.65;

    /**
     * 融合评分相对截断因子（0~1）。
     * 融合后只保留 fusedScore ≥ top1 × factor 的结果。
     * 用于拦截「向量未命中但 BM25 关键词碰巧命中」的漏网之鱼。
     * 默认 0.75。
     */
    private double relativeCutoffFactor = 0.75;

    // getters / setters ...
}
```

### SearchService.java（三处改动）

**改动1**：pgvector 召回加阈值——`similarityThresholdAll()` → `similarityThreshold(配置)`

```java
// 旧代码
var searchRequest = SearchRequest.builder()
        .query(query).topK(topK)
        .similarityThresholdAll()   // ← 不做任何过滤
        .build();

// 新代码
var searchRequest = SearchRequest.builder()
        .query(query).topK(topK)
        .similarityThreshold(config.getSimilarityThreshold())  // ← HNSW 层拦截
        .build();
```

**改动2**：搜索方法末加相对分差截断

```java
// 排序后、取 topK 前
results.sort(Comparator.comparingDouble(SearchResultItem::fusedScore).reversed());

// 相对分差截断：不到第一名 75% 的不要
if (!results.isEmpty()) {
    double cutoff = results.get(0).fusedScore() * config.getRelativeCutoffFactor();
    results = results.stream()
            .filter(r -> r.fusedScore() >= cutoff)
            .collect(Collectors.toList());
}

List<SearchResultItem> topResults = results.stream().limit(topK).toList();
```

**改动3**：`findRelated()` 同理加截断

```java
results.sort(Comparator.comparingDouble(SearchResultItem::fusedScore).reversed());

if (!results.isEmpty()) {
    double cutoff = results.get(0).fusedScore() * config.getRelativeCutoffFactor();
    results = results.stream()
            .filter(r -> r.fusedScore() >= cutoff)
            .collect(Collectors.toList());
}
return results.stream().limit(topK).toList();
```

### application.yml（可调参）

```yaml
mindcache:
  search:
    similarity-threshold: 0.65    # 调高→更严格；调低→更宽松
    relative-cutoff-factor: 0.75  # 调高→截断更狠；调低→更宽容
```

搜 "spring" — 调参后最终效果：

| 改前（无过滤） | 第一次上线（threshold=0.65） | 调参后（threshold=0.3, cutoff=0.75） |
|------|------|------|

```

最终：    2 条 → Spring Boot 3.5(0.95) / Spring AI(0.73)
```

## 面试口述版

> "这个问题暴露了纯 topK 截断的缺陷——当库里笔记数量少于 topK 时，即使完全不相关的结果也会因为没有任何过滤机制而被返回。
>
> 我的方案是**双重过滤**，只加了大约 15 行代码：
>
> **第一道在 pgvector 索引层**：用 Spring AI 的 `similarityThreshold` 参数，让 HNSW 索引在扫描时就跳过不相关的向量——这是数据库侧过滤，效率高。
>
> **第二道在融合层**：加了一个相对分差截断。不是写死 0.3 这种绝对值，而是用 top1 的分数作为基准——fusedScore 不到第一名 75% 的，视为弱相关。这样既能在'大家都高分'时不误杀，又能在'大家都不高'时保住最好的几条。
>
> 两个参数都在 yml 里可调，评测时不用改代码。验证结果：搜 'spring' 从之前返回 4 条（含无关的 '12'）变成只返回 2 条真正相关的（Spring Boot 3.5 和 Spring AI）。

## 追加调参记录：pgvector 阈值过严问题的发现与修正

### 现场追问

> "但是 Spring Boot 3.5 跟 Spring Boot 虚拟线程也是 spring 相关的东西啊，为什么也被过滤了？"

### 问题

第一次上线时 `similarityThreshold=0.65`，验证结果搜 "spring" 只返回 1 条（Spring AI）。**Spring Boot 3.5（fusedScore 排第一 0.88）被 pgvector 阈值误杀**。

根因是向量阈值一刀切，无法区分「中文乱码导致向量差」和「真的不相关」。Spring Boot 3.5 这条纯文本笔记的向量和 "spring" 的余弦相似度实际上是够的——问题出在 Spring AI 的 pgvector `similarityThreshold` 映射不是我以为的线性关系，0.65 在实际运行中比预期严格得多。

### 修正

将 `similarityThreshold` 从 0.65 下调到 **0.3**，让 pgvector 这道只做「粗筛」——拦截余弦相似度极低（< 0.3）的明显无关结果，"12" 这种。真正的精筛交给第二道相对分差截断。

### 最终参数

```yaml
mindcache:
  search:
    similarity-threshold: 0.3      # 粗筛：只拦明显无关的
    relative-cutoff-factor: 0.75   # 精筛：top1 的 75% 以下是弱相关
```

两道分工：

```
pgvector 0.3  → 拦截：12（余弦相似度 ≈ 0.25）
               放行：Spring Boot 3.5(0.89)、Spring AI(1.0)、虚拟线程(0.36)

相对截断 0.75 → 拦截：虚拟线程（0.38 < 0.95×0.75 = 0.71）
               放行：Spring Boot 3.5(0.95)、Spring AI(0.73)
```

### 修正后搜 "spring" 的最终结果

| 排名 | 笔记 | fusedScore | 向量 | BM25 | 时间衰减 |
|------|------|-----------|------|------|---------|
| #1 | Spring Boot 3.5 | **0.95** | 0.89 | 1.0 | 1.0 |
| #2 | Spring AI 相关知识 | **0.73** | 1.0 | 0.10 | 1.0 |
| — | 虚拟线程 | 0.38 | — | — | 相对截断过滤 |
| — | 12 | 0.20 | — | — | pgvector 0.3 拦截 |

### 「虚拟线程」被过滤的数学

```
虚拟线程 fusedScore = 0.38
第一名 Spring Boot 3.5 = 0.95

截断线 = 0.95 × 0.75 = 0.71
0.38 < 0.71  →  过滤 ❌
```

拆开它的三分：

| 信号 | 值 | 原因 |
|------|-----|------|
| 向量语义 | 0.36 | 内容侧重"学习虚拟线程很开心"而非 "Spring Boot" 本身，语义距离较远 |
| BM25 关键词 | 0.0 | AUDIO 类型经去口语化清洗后，Lucene 关键词匹配度低 |
| 时间衰减 | 0.97 | 创建于 4 天前，几乎未衰减 |

它和 "spring" 本质上是**弱语义相关**——虚拟线程确实属于 Spring Boot 生态，但笔记的表述方式让向量模型没给出高分。如果要放行它，把 `relativeCutoffFactor` 降到 0.4（截断线 = 0.95×0.4 = 0.38）即可。

### 启示

> "0.65 这个数是我从文档里拍脑袋定的。实际跑下来发现，Spring AI 的 pgvector 阈值映射不是线性关系，0.65 在真实数据上比预期严得多。这个经历让我意识到：检索参数没有一个标准答案——必须用真实数据跑几轮，看实际过滤效果来调。好在参数都在 yml 里，不用改代码。"
